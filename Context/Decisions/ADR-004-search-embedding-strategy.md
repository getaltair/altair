# ADR-004: Search Index and Embedding Strategy

## Status

Accepted

## Date

2026-04-12

## Context

Altair requires cross-domain search spanning Guidance, Knowledge, and Tracking. Users need to find quests, notes, items, and their relationships from a single search interface. The architecture spec describes a hybrid search pipeline combining keyword full-text search with semantic embedding search.

Key constraints:
- v1 must run on a Raspberry Pi 4 (4GB RAM) — no local GPU
- AI features are v1 scope but optional for users (graceful degradation)
- Self-hosted deployment — no mandatory external service dependency
- Search must work without AI/embeddings configured (keyword search is the baseline)

## Decision

### v1: PostgreSQL Full-Text Search + External Embeddings

**Keyword search (always available):**
- PostgreSQL built-in full-text search (`tsvector` / `tsquery`)
- Denormalized search documents indexed via async background jobs
- Cross-domain results: each result preserves entity type, app origin, initiative context, preview snippet, relevance score
- Zero additional infrastructure — uses the existing Postgres instance

**Semantic search (when configured):**
- `pgvector` extension for vector storage and similarity search
- Embeddings generated by an **external embedding service** (OpenAI-compatible API)
- Provider abstraction layer: adapter pattern supporting OpenAI, Anthropic, Ollama, or any OpenAI-compatible endpoint
- Embedding jobs run asynchronously in the background worker
- Hybrid ranking combines keyword FTS score + vector similarity + domain boosts

**Search pipeline:**
```
Source → Extract text → FTS index (tsvector)
                     → Embedding job (async, external API) → pgvector index
Query → Keyword results + Semantic results → Hybrid ranking → Results
```

### Graceful Degradation

| Configuration | Search Capability |
|--------------|-------------------|
| No AI configured | Keyword FTS only (fully functional) |
| External embedding API configured | Keyword + semantic hybrid search |
| Embedding API unavailable (transient) | Falls back to keyword FTS, queues embedding jobs for retry |

AI features never block core CRUD or search. If the embedding service is down, keyword search continues uninterrupted.

### AI Feature Scope (v1)

The following capabilities use the same provider abstraction:
- Note summarization
- Quest extraction from notes
- Suggested entity links
- Semantic enrichment of search index

All are optional, user-triggered (not proactive), and degrade cleanly if no provider is configured.

### Deferred to Post-v1

- Local embeddings via Ollama or on-device inference
- GPU-accelerated search
- Proactive AI suggestions
- OCR/transcription pipelines
- Fine-tuning or domain adaptation

## Consequences

### Positive

- Zero additional infrastructure for baseline search (Postgres only)
- pgvector is a Postgres extension, not a separate service — no new container
- External embedding API keeps compute off the Pi
- Provider abstraction avoids vendor lock-in
- Search works fully without AI configured

### Negative

- External embedding API adds latency to embedding generation (acceptable for async background jobs)
- Users without API access get keyword-only search (good but not semantic)
- pgvector on Pi 4 may have limited performance with large vector indexes — acceptable for personal/household scale
- Embedding costs depend on external provider pricing

### Neutral

- Hybrid ranking weights (keyword vs. semantic vs. domain boost) will need tuning during implementation
- The provider abstraction adds a layer of indirection but is necessary for the multi-provider requirement
