## Key Architectural Decisions

### 1. Offline-First with Dual IDs

**Decision:** Use client-generated `id` as primary key, with nullable `serverId` for backend sync.

```dart
TextColumn get id => text().clientDefault(() => const Uuid().v4())();
TextColumn get serverId => text().nullable()();
```

**Why:**
- Tasks can be created instantly offline without waiting for server
- No data loss if network fails
- Clean sync strategy: local ID never changes, server ID added on sync

**ADHD Benefit:** Instant task capture prevents "forgot what I was going to add" syndrome

### 2. IntEnum vs TextEnum

**Decision:** Use `intEnum<TaskState>()` instead of `textEnum<TaskState>()`

```dart
// ✅ Better performance
IntColumn get state => intEnum<TaskState>().withDefault(Constant(TaskState.inbox.index))();

// ❌ Slower, larger storage
TextColumn get state => textEnum<TaskState>()...
```

**Why:**
- Integers are faster to index and query
- Smaller database size (4 bytes vs variable string length)
- Type-safe at compile time (Drift 2.28+ feature)

**Trade-off:** Slightly less readable in raw SQL queries, but the performance gain is worth it for ADHD users who need instant UI responsiveness.

### 3. JSON Type Converter for ADHD Profiles

**Decision:** Store ADHD preferences as JSON in SQLite

```dart
class JsonTypeConverter extends TypeConverter<Map<String, dynamic>, String> {
  // Converts Map ↔ JSON string for storage
}

TextColumn get adhdProfile => text()
    .map(const JsonTypeConverter())
    .clientDefault(() => '{}')();
```

**Why:**
- Flexible schema (users can have different preference sets)
- No migration needed when adding new ADHD features
- Easy to extend with new sensory preferences, focus patterns, etc.

**ADHD Benefit:** Each user can customize their experience without rigid schema constraints

### 4. Sync Metadata in BaseModel

**Decision:** Every table includes sync tracking fields via mixin

```dart
mixin BaseModel on Table {
  TextColumn get id => ...
  TextColumn get serverId => ...
  DateTimeColumn get lastSyncedAt => ...
  IntColumn get version => ...  // Optimistic locking
  BoolColumn get pendingSync => ...
}
```

**Why:**
- Consistent sync strategy across all tables
- Version number enables conflict resolution
- `pendingSync` flag makes background sync trivial: just query where `pendingSync = true`

**Key Insight:** This pattern makes the sync engine simple - it doesn't need to track state separately.

### 5. ADHD-Specific Indexes

**Decision:** Add composite indexes for common ADHD user workflows

```dart
Index('idx_tasks_user_state',
  'CREATE INDEX idx_tasks_user_state ON tasks(user_id, state)'),

Index('idx_tasks_pending_sync',
  'CREATE INDEX idx_tasks_pending_sync ON tasks(pending_sync) WHERE pending_sync = 1'),
```

**Why:**
- "Show me my inbox" query is instant (user_id + state index)
- Background sync is fast (pending_sync partial index)
- Parent-child task decomposition is performant (parent_id index)

**ADHD Benefit:** No waiting = no forgetting what you were doing

## Critical Learnings from Agent Reviews

### Issue 1: DateTime.now Syntax Error

**Problem:**
```dart
// ❌ WRONG - This doesn't compile in Drift 2.28
DateTimeColumn get createdAt => dateTime().clientDefault(DateTime.now)();
```

**Fix:**
```dart
// ✅ CORRECT - clientDefault needs a function
DateTimeColumn get createdAt => dateTime().clientDefault(() => DateTime.now())();
```

**Lesson:** `clientDefault` expects a **function** that returns the value, not the value itself. This allows Drift to call it fresh each time a row is inserted.

### Issue 2: IntEnum Companion Type Change

**Problem:**
```dart
// ❌ WRONG in Drift 2.28 - Companion expects enum, not int
TasksCompanion(state: Value(newState.index))
```

**Fix:**
```dart
// ✅ CORRECT - Pass the enum value directly
TasksCompanion(state: Value(newState))
```

**Lesson:** Drift 2.28+ changed how intEnum works. The `Companion` classes now handle the int conversion automatically, so you work with the enum type directly for better type safety.

### Issue 3: Enum Filtering Syntax

**Problem:**
```dart
// ❌ WRONG - equals() doesn't work with enums in queries
where((t) => t.state.equals(state.index))
```

**Fix:**
```dart
// ✅ CORRECT - Use equalsValue for enum columns
where((t) => t.state.equalsValue(state))
```

**Lesson:** For intEnum columns, use `.equalsValue()` in queries, not `.equals()`. Drift provides the right method for type-safe enum filtering.

### Issue 4: Missing Primary Key Declaration

**Problem:**
```dart
class BaseModel extends Table {
  TextColumn get id => ...
  // Missing: @override Set<Column> get primaryKey => {id};
}
```

**Fix:**
```dart
mixin BaseModel on Table {
  TextColumn get id => ...

  @override
  Set<Column> get primaryKey => {id};  // ✅ Required!
}
```

**Lesson:** Drift tables MUST explicitly declare their primary key. It doesn't assume `id` is the primary key even if that's a convention.

## Query Method Patterns

### Reactive Streams (watch)

Use when UI needs to automatically update:

```dart
Stream<List<Task>> watchAllTasks(String userId) {
  return (select(tasks)
    ..where((t) => t.userId.equals(userId))
    ..orderBy([(t) => OrderingTerm.desc(t.createdAt)]))
    .watch();  // ← Returns Stream
}
```

**Usage in UI:**
```dart
ref.watch(tasksProvider)  // Rebuilds widget when tasks change
```

**ADHD Benefit:** Task list automatically updates when you add/edit tasks from another screen. No manual refresh = less cognitive load.

### One-Time Queries (get)

Use for non-UI operations:

```dart
Future<List<Task>> getUnsyncedTasks() {
  return (select(tasks)
    ..where((t) => t.pendingSync.equals(true)))
    .get();  // ← Returns Future
}
```

**Usage in Sync Service:**
```dart
final pending = await db.getUnsyncedTasks();
for (final task in pending) {
  await api.syncTask(task);
}
```

### State Transitions

Special method for ADHD workflow:

```dart
Future<bool> transitionTaskState(String id, TaskState newState) async {
  final update = TasksCompanion(
    state: Value(newState),
    completedAt: newState == TaskState.done
      ? Value(DateTime.now())  // ✅ Auto-timestamp on completion
      : const Value.absent(),
    pendingSync: const Value(true),  // Mark for sync
  );
  return updateTask(id, update);
}
```

**Why Special Method:**
- Handles completion timestamp automatically
- Marks for sync
- Single place to add celebration triggers later

## Performance Optimizations

### 1. Partial Indexes

```sql
CREATE INDEX idx_tasks_pending_sync
  ON tasks(pending_sync)
  WHERE pending_sync = 1
```

Only indexes rows that need syncing. Much faster than full index when 99% of tasks are already synced.

### 2. Composite Indexes

```sql
CREATE INDEX idx_tasks_user_state
  ON tasks(user_id, state)
```

Optimizes the most common query: "Show me my active tasks"

### 3. Cognitive Load Index

```sql
CREATE INDEX idx_tasks_state_load
  ON tasks(state, cognitive_load)
```

Enables fast sorting: "Show me easy tasks I can knock out quickly"

1. **Sync Metadata**

   - Mark synced → verify pendingSync = false
   - Update task → verify pendingSync = true again

2. **State Transitions**

## Common Pitfalls to Avoid

1. **Don't use .equals() with enums** → Use .equalsValue()
2. **Don't forget @override primaryKey** → Tables won't compile
3. **Don't use .index in Companions** → Pass enum directly
4. **Don't skip version field** → Needed for conflict resolution
5. **Don't forget foreignKey cascade** → Or you'll have orphaned tasks
