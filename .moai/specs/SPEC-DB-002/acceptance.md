## FR-001: SQLDelight Dependency Integration

### Scenario 1.1: Android Target Builds with SQLDelight

```gherkin
Given SQLDelight version 2.0.2 is added to gradle/libs.versions.toml
And the SQLDelight plugin is applied in shared/build.gradle.kts
And sqldelight-android-driver is added to androidMain dependencies
When the Android target is built using ./gradlew shared:compileKotlinAndroid
Then the build completes without errors
And no dependency resolution failures occur
```

### Scenario 1.2: iOS Target Builds with SQLDelight

```gherkin
Given SQLDelight version 2.0.2 is added to gradle/libs.versions.toml
And the SQLDelight plugin is applied in shared/build.gradle.kts
And sqldelight-native-driver is added to iosMain dependencies
When the iOS target is built using ./gradlew shared:compileKotlinIosArm64
Then the build completes without errors
And NativeSqliteDriver class is available in iosMain source set
```

### Scenario 1.3: Version Catalog Entries Correct

```gherkin
Given the gradle/libs.versions.toml file exists
When inspecting the versions section
Then sqldelight version is defined as "2.0.2"
And sqldelight-runtime library alias references the sqldelight version
And sqldelight-android-driver library alias references the sqldelight version
And sqldelight-native-driver library alias references the sqldelight version
```

## FR-002: Schema Definition

### Scenario 2.1: Quest Entity Class Generated

```gherkin
Given Quest.sq is defined in shared/src/commonMain/sqldelight/
And the schema includes id, title, description, status, epic_id, created_at, updated_at, deleted_at, sync_version columns
When SQLDelight generates code using ./gradlew shared:generateCommonMainAltairDatabaseInterface
Then Quest entity class is created in com.getaltair.altair.database package
And Quest class contains properties for all defined columns
And property types match: id(String), title(String), description(String?), status(String), epic_id(String?), created_at(String), updated_at(String), deleted_at(String?), sync_version(Long)
```

### Scenario 2.2: Note Queries Generated

```gherkin
Given Note.sq is defined with CREATE TABLE and query definitions
When SQLDelight generates code
Then NoteQueries class is created
And NoteQueries contains selectAll() method returning Query<Note>
And NoteQueries contains selectById(id: String) method returning Query<Note>
And NoteQueries contains selectByFolderId(folder_id: String) method returning Query<Note>
And NoteQueries contains insert(...) method
And NoteQueries contains updateById(...) method
And NoteQueries contains softDeleteById(...) method
```

### Scenario 2.3: Item Queries Generated

```gherkin
Given Item.sq is defined with CREATE TABLE and query definitions
When SQLDelight generates code
Then ItemQueries class is created
And ItemQueries contains selectAll() method
And ItemQueries contains selectById(id: String) method
And ItemQueries contains selectByLocationId(location_id: String) method
And ItemQueries contains selectByContainerId(container_id: String) method
And ItemQueries contains insert(...) method
And ItemQueries contains updateById(...) method
And ItemQueries contains softDeleteById(...) method
```

### Scenario 2.4: Indexes Created for Performance

```gherkin
Given Quest.sq, Note.sq, and Item.sq contain index definitions
When the database schema is applied
Then idx_quest_status index exists on Quest(status) with partial index WHERE deleted_at IS NULL
And idx_quest_epic index exists on Quest(epic_id) with partial index WHERE deleted_at IS NULL
And idx_note_folder index exists on Note(folder_id) with partial index WHERE deleted_at IS NULL
And idx_item_location index exists on Item(location_id) with partial index WHERE deleted_at IS NULL
And idx_item_container index exists on Item(container_id) with partial index WHERE deleted_at IS NULL
```

## FR-003: Type-Safe Query Generation

### Scenario 3.1: Insert Query Type Safety

```gherkin
Given QuestQueries class is generated by SQLDelight
When calling questQueries.insert()
Then the method requires parameters: id(String), title(String), description(String?), status(String), epic_id(String?), created_at(String), updated_at(String), deleted_at(String?), sync_version(Long)
And compilation fails if required non-null parameters are omitted
And compilation fails if parameter types do not match
```

### Scenario 3.2: Select Query Return Types

```gherkin
Given QuestQueries class is generated by SQLDelight
When calling questQueries.selectById(id).executeAsOneOrNull()
Then return type is Quest? (nullable)
And null is returned when no record matches the ID
When calling questQueries.selectAll().executeAsList()
Then return type is List<Quest>
And empty list is returned when no records exist
```

### Scenario 3.3: Update Query Parameter Validation

```gherkin
Given QuestQueries class is generated by SQLDelight
When calling questQueries.updateById()
Then the method requires all update parameters plus the id parameter
And the id parameter identifies the record to update
And compilation fails if parameter types do not match schema
```

## FR-004: Repository Implementation

### Scenario 4.1: Quest Repository Create Operation

```gherkin
Given QuestRepository is initialized with database connection
And a Quest object with title="Test Quest" and status="pending"
When calling repository.create(quest)
Then a new record is inserted with generated ULID as id
And the ULID is 26 characters in Crockford Base32 format
And created_at is set to current UTC timestamp in ISO 8601 format
And updated_at is set to current UTC timestamp in ISO 8601 format
And sync_version is set to 0
And the created Quest with id is returned
```

### Scenario 4.2: Quest Repository Soft Delete

```gherkin
Given QuestRepository is initialized with database connection
And a Quest record exists with id="01HYD3XVKW8JN4GPBC1YZF9R2T"
When calling repository.delete("01HYD3XVKW8JN4GPBC1YZF9R2T")
Then the record's deleted_at is set to current UTC timestamp
And the record's updated_at is updated to current UTC timestamp
And the record is NOT permanently removed from database
And repository.findById("01HYD3XVKW8JN4GPBC1YZF9R2T") returns null
And the record still exists when querying without soft-delete filter
```

### Scenario 4.3: Note Repository Find By Folder

```gherkin
Given NoteRepository is initialized with database connection
And three Note records exist:
  | id | title | folder_id |
  | A  | Note1 | folder1   |
  | B  | Note2 | folder1   |
  | C  | Note3 | folder2   |
When calling repository.findByFolderId("folder1")
Then a list of 2 notes is returned
And all returned notes have folder_id="folder1"
And notes are ordered by created_at descending
```

### Scenario 4.4: Item Repository Find By Location and Container

```gherkin
Given ItemRepository is initialized with database connection
And Item records exist with various location_id and container_id values
When calling repository.findByLocationId("location1")
Then only items with location_id="location1" and deleted_at IS NULL are returned
When calling repository.findByContainerId("container1")
Then only items with container_id="container1" and deleted_at IS NULL are returned
```

## Edge Case Scenarios

### Scenario E1: Database Not Initialized

```gherkin
Given DatabaseProvider.initialize() has NOT been called
When any repository method is invoked
Then IllegalStateException is thrown with message "Database not initialized"
And the application does not crash unexpectedly
```

### Scenario E2: Concurrent Read Operations

```gherkin
Given the database contains 100 Quest records
When 10 concurrent read operations are executed
Then all operations complete successfully
And no SQLite locking errors occur
And all returned data is consistent
```

### Scenario E3: Large Data Set Query Performance

```gherkin
Given the Quest table contains 10,000 records
When repository.findAll() is executed
Then the query completes in under 500ms
And memory usage remains within acceptable bounds
```

### Scenario E4: Invalid ULID Handling

```gherkin
Given a findById() call with invalid ID format "invalid-id-format"
When the query executes
Then null is returned (not found)
And no exception is thrown
And no database corruption occurs
```

### Scenario E5: Null Description and Optional Fields

```gherkin
Given a Quest is created with description=null and epic_id=null
When the record is persisted and retrieved
Then description is null in the returned Quest
And epic_id is null in the returned Quest
And no "NOT NULL constraint failed" error occurs
```

### Scenario E6: Empty String vs Null Handling

```gherkin
Given a Quest is created with description="" (empty string)
When the record is persisted and retrieved
Then description equals "" (empty string), not null
And the distinction between empty string and null is preserved
```

## Non-Functional Requirement Verification

### NFR-001: Query Performance Under 50ms

```gherkin
Given the database is initialized with AndroidSqliteDriver or NativeSqliteDriver
And single Quest, Note, and Item records exist
When timing insert operation for a single record
Then execution time is under 50ms
When timing select by ID operation
Then execution time is under 50ms
When timing update operation for a single record
Then execution time is under 50ms
When timing soft-delete operation
Then execution time is under 50ms
```

### NFR-003: Thread-Safe Database Access

```gherkin
Given the database is initialized
When multiple coroutines simultaneously access the database
Then no SQLite thread-safety violations occur
And no data corruption is observed
And all operations complete successfully
```

### NFR-004: Cross-Platform Query Consistency

```gherkin
Given identical test data is created on Android and iOS
When selectAll() is executed on Android
And selectAll() is executed on iOS
Then the returned record count is identical
And the record field values are identical
And timestamp formats are consistent (ISO 8601 UTC)
```
