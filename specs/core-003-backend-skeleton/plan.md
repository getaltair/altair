# Implementation Plan: Tauri Backend Service Foundation

**Branch**: `spec/core-003-backend-skeleton` | **Date**: 2025-12-06 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/core-003-backend-skeleton/spec.md`

## Summary

Build the foundational Rust backend skeleton for all Tauri desktop apps (Guidance, Knowledge, Tracking). This includes:

- **AppState** with embedded SurrealDB connection and runtime configuration
- **ApiError** type with SCREAMING_SNAKE_CASE codes for IPC error handling
- **health_check** command returning version, database status, and timestamp
- **Configuration** loading from TOML with sensible defaults
- **Logging** infrastructure using tracing with JSON output and 7-day rotation
- **Type generation** via tauri-specta for TypeScript bindings

## Technical Context

**Language/Version**: Rust 2024 edition (1.91.1+)
**Primary Dependencies**: Tauri 2.9.3, SurrealDB 2.x (surrealkv), tauri-specta 2.0.0-rc.20, tracing
**Storage**: SurrealDB embedded (surrealkv storage engine)
**Testing**: cargo test
**Target Platform**: Linux, macOS, Windows (desktop apps)
**Project Type**: Monorepo with Cargo workspace + pnpm workspaces
**Performance Goals**: Backend startup < 2s, health_check < 50ms
**Constraints**: Single-user desktop focus, no multi-tenancy
**Scale/Scope**: 3 desktop apps sharing backend crates

## Implementation Phases

### Phase 1: Core Types and Configuration

**Goal**: Establish ApiError and AppConfig types in altair-core

**Tasks**:

1. **Add ApiError type** (`backend/crates/altair-core/src/api_error.rs`)

   - Struct with `code: String`, `message: String`, `details: Option<serde_json::Value>`
   - SCREAMING_SNAKE_CASE codes (e.g., `DB_CONNECTION_FAILED`)
   - Implement `serde::Serialize` for IPC transport
   - Implement `From<altair_core::Error>` for conversion
   - Add `specta::Type` derive for TypeScript generation

2. **Add AppConfig type** (`backend/crates/altair-core/src/config.rs`)

   - Struct with `database_path: PathBuf`, `log_level: LogLevel`, `log_dir: PathBuf`, `log_retention_days: u8`
   - Default values: INFO log level, 7-day retention
   - TOML deserialization support
   - Platform-aware default paths (`~/.config/altair/` on Linux)

3. **Update Cargo.toml dependencies**

   - Add `directories` crate for platform paths
   - Add `toml` for config parsing
   - Add `specta` with derive feature

4. **Export new types from lib.rs**

**Exit Criteria**: `cargo test -p altair-core` passes, types are exported

---

### Phase 2: Database Connection Layer

**Goal**: Implement real SurrealDB embedded connection in altair-db

**Tasks**:

1. **Enable SurrealDB dependency** (`backend/crates/altair-db/Cargo.toml`)

   - Uncomment surrealdb with `kv-surrealkv` feature
   - Add tokio with required features

2. **Create connection module** (`backend/crates/altair-db/src/connection.rs`)

   - `SurrealConnection` struct wrapping `Surreal<Db>`
   - `connect(config: &DatabaseConfig) -> Result<Self>`
   - Use surrealkv for embedded storage
   - Implement `DatabaseClient` trait

3. **Create health check module** (`backend/crates/altair-db/src/health.rs`)

   - `check_database_health(conn: &SurrealConnection) -> Result<DatabaseHealth>`
   - Returns connection status, response time

4. **Update lib.rs exports**

**Exit Criteria**: `cargo test -p altair-db` passes, connection can be established

---

### Phase 3: Logging Infrastructure

**Goal**: Set up tracing with JSON output and log rotation

**Tasks**:

1. **Add logging dependencies** (workspace Cargo.toml)

   - `tracing-subscriber` with JSON feature
   - `tracing-appender` for file output and rotation

2. **Create logging setup function** (`backend/crates/altair-core/src/logging.rs`)

   - `init_logging(config: &AppConfig) -> Result<LogGuard>`
   - JSON format for file output
   - Human-readable for console in debug mode
   - Daily rotation with 7-day retention

3. **Return guard to prevent log handle drop**

**Exit Criteria**: Logs written to configured directory in JSON format

---

### Phase 4: AppState and Command Infrastructure

**Goal**: Create AppState and health_check command in one app (guidance)

**Tasks**:

1. **Create AppState module** (`apps/guidance/src-tauri/src/state.rs`)

   - Struct with `db: SurrealConnection`, `config: AppConfig`
   - `AppState::new(config: AppConfig) -> Result<Self>`
   - Initializes DB connection, sets up logging

2. **Create health_check command** (`apps/guidance/src-tauri/src/commands/health.rs`)

   - `#[tauri::command]` with `#[specta::specta]`
   - Returns `ApiResult<HealthStatus>`
   - Queries DB health, includes version from Cargo.toml

3. **Update lib.rs**

   - Import state and commands
   - Initialize AppState in `setup` closure
   - Register health_check in invoke_handler
   - Configure tauri-specta export path

4. **Test command invocation**

**Exit Criteria**: `cargo build -p guidance` succeeds, health_check command registered

---

### Phase 5: Replicate to All Apps

**Goal**: Copy pattern to knowledge, tracking, and mobile apps

**Tasks**:

1. **Copy state.rs and commands/ to each app**

   - `apps/knowledge/src-tauri/src/`
   - `apps/tracking/src-tauri/src/`
   - `apps/mobile/src-tauri/src/`

2. **Update each app's lib.rs**

   - Same pattern as guidance

3. **Update Cargo.toml for each app**
   - Add altair-db dependency

**Exit Criteria**: All four apps build and register health_check

---

### Phase 6: TypeScript Binding Generation

**Goal**: Generate and validate TypeScript bindings

**Tasks**:

1. **Create bindings generation workflow**

   - Each app exports to `packages/bindings/src/{app}.ts`
   - Ensure bindings include `health_check`, `HealthStatus`, `ApiError`

2. **Run generation**

   - Build each app in debug mode to trigger specta export
   - Verify generated files

3. **Add TypeScript index.ts re-exports**

   - Export common types from all app bindings

4. **Validate TypeScript compilation**
   - `pnpm --filter @altair/bindings build`

**Exit Criteria**: TypeScript bindings compile without errors

---

### Phase 7: Integration Testing

**Goal**: Verify end-to-end functionality

**Tasks**:

1. **Write integration tests**

   - Backend startup timing (< 2s)
   - health_check command invocation
   - Configuration loading from defaults

2. **Test on all platforms** (CI)

   - Linux, macOS, Windows

3. **Document test procedures**

**Exit Criteria**: All acceptance criteria from spec pass
