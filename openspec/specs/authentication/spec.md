## Requirements
### Requirement: Password Hashing
The system SHALL hash user passwords using Argon2id before storage.

#### Scenario: Password hashed on registration
- **WHEN** a user registers with a password
- **THEN** the password is hashed using Argon2id with memory=64MB, iterations=3, parallelism=4
- **AND** only the hash is stored in the database

#### Scenario: Password verified on login
- **WHEN** a user attempts to login with a password
- **THEN** the system verifies the password against the stored Argon2id hash
- **AND** timing-safe comparison is used to prevent timing attacks

### Requirement: JWT Access Tokens
The system SHALL issue JWT access tokens for authenticated API access.

#### Scenario: Access token issued on login
- **WHEN** a user successfully authenticates
- **THEN** the system issues a JWT access token with 15-minute expiration
- **AND** the token contains claims: sub (userId), role, email, aud, iss, iat, exp

#### Scenario: Access token validated on request
- **WHEN** a client makes an authenticated request
- **THEN** the server validates the JWT signature using HMAC-SHA256
- **AND** the server verifies the token has not expired
- **AND** the server extracts user claims for authorization

#### Scenario: Invalid access token rejected
- **WHEN** a client presents an invalid or expired access token
- **THEN** the server returns 401 Unauthorized
- **AND** the response indicates the token is invalid or expired

### Requirement: Refresh Token Rotation
The system SHALL support refresh tokens for obtaining new access tokens without re-authentication.

#### Scenario: Refresh token issued on login
- **WHEN** a user successfully authenticates
- **THEN** the system issues a refresh token with 30-day expiration
- **AND** the refresh token is stored (hashed) in the database with device metadata

#### Scenario: Access token refreshed
- **WHEN** a client calls refresh with a valid refresh token
- **THEN** the system issues a new access token
- **AND** the system issues a new refresh token (rotation)
- **AND** the old refresh token is invalidated

#### Scenario: Refresh token revoked on logout
- **WHEN** a user logs out
- **THEN** the current refresh token is revoked
- **AND** subsequent refresh attempts with that token fail

### Requirement: Invite-Only Registration
The system SHALL require invite codes for new user registration by default.

#### Scenario: Registration with valid invite code
- **WHEN** a new user registers with a valid, unused invite code
- **THEN** the user account is created
- **AND** the invite code is marked as used
- **AND** authentication tokens are returned

#### Scenario: Registration without invite code rejected
- **WHEN** a new user attempts to register without an invite code
- **AND** at least one user exists in the system
- **THEN** the registration is rejected with an appropriate error

#### Scenario: First user bootstrap
- **WHEN** the first user registers
- **AND** no users exist in the system
- **THEN** the user is created with admin role without requiring an invite code

#### Scenario: Invite code generated by admin
- **WHEN** an admin user requests a new invite code
- **THEN** the system generates a unique invite code with configurable expiration
- **AND** the invite code is associated with the creating admin

### Requirement: Session Revocation
The system SHALL support revoking user sessions for security.

#### Scenario: Single session revoked
- **WHEN** a user logs out
- **THEN** only the current session's refresh token is revoked
- **AND** other sessions remain active

#### Scenario: All sessions revoked
- **WHEN** a user requests to revoke all sessions
- **THEN** all refresh tokens for that user are revoked
- **AND** the user must re-authenticate on all devices

### Requirement: Password Change
The system SHALL allow authenticated users to change their password.

#### Scenario: Password changed successfully
- **WHEN** an authenticated user provides their current password and a new password
- **AND** the current password is correct
- **THEN** the password is updated to the new value
- **AND** all existing sessions remain valid

#### Scenario: Password change with incorrect current password
- **WHEN** an authenticated user provides an incorrect current password
- **THEN** the password change is rejected
- **AND** an appropriate error is returned

### Requirement: Secure Token Storage (Client)
The system SHALL store authentication tokens securely on each client platform.

#### Scenario: Android token storage
- **WHEN** the Android client stores authentication tokens
- **THEN** tokens are stored using EncryptedSharedPreferences (Jetpack Security)
- **AND** tokens are encrypted with AES-256

#### Scenario: iOS token storage
- **WHEN** the iOS client stores authentication tokens
- **THEN** tokens are stored using Keychain Services
- **AND** tokens are protected by device security settings

#### Scenario: Desktop token storage
- **WHEN** the desktop client stores authentication tokens
- **THEN** tokens are stored using the system keyring or Java KeyStore
- **AND** tokens are encrypted at rest

### Requirement: Auth Context Injection
The system SHALL inject authenticated user context into request handlers.

#### Scenario: User context available in repositories
- **WHEN** a repository method executes during an authenticated request
- **THEN** the AuthContext provides the current user's ID
- **AND** all queries are automatically scoped to that user

#### Scenario: Unauthenticated request rejected
- **WHEN** a request to a protected endpoint lacks valid authentication
- **THEN** the request is rejected with 401 Unauthorized
- **AND** no repository methods are executed

### Requirement: JWT Configuration
The system SHALL load JWT configuration from environment variables.

#### Scenario: JWT secret from environment
- **WHEN** the server starts
- **THEN** the JWT signing secret is loaded from ALTAIR_JWT_SECRET environment variable
- **AND** the server fails to start if the secret is not configured

#### Scenario: Default token expiration configurable
- **WHEN** the server starts
- **THEN** access token expiration is loaded from ALTAIR_JWT_ACCESS_EXPIRY (default: 900 seconds)
- **AND** refresh token expiration is loaded from ALTAIR_JWT_REFRESH_EXPIRY (default: 2592000 seconds)
